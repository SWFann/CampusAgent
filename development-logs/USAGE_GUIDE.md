# 开发日志系统使用指南

## 🎯 核心原则

1. **透明性** - 每个任务都留有完整记录
2. **可追溯** - 可以重建整个开发过程
3. **实时性** - 边做边记，不事后补记
4. **诚实性** - 如实记录问题、延期和阻塞

## 📁 目录说明

### completed/ - 已完成任务

存放已经完成并通过验收的任务日志。

**命名**：`{阶段}-{序号}-{简短描述}.md`

**示例**：
- `P0-01-domain-vocabulary.md`
- `P3-04-login-implementation.md`
- `P9-12-meal-scene-api.md`

### in-progress/ - 进行中的任务

存放当前正在开发的任务日志。

**规则**：
- 同一时间只能有一个任务在此目录
- 完成任务后立即移动到 `completed/`
- 如果阻塞，在日志中标记阻塞原因

### planned/ - 计划中的任务

存放已规划但尚未开始的任务日志（可选）。

**用途**：
- 提前规划复杂任务的框架
- 记录依赖关系和注意事项
- 作为任务启动前的准备

### templates/ - 模板

存放任务日志的标准模板。

**使用**：复制 `task-template.md` 到对应目录，填写任务ID和基本信息。

## 📝 日志模板详解

### Frontmatter（元数据）

```yaml
---
task_id: P0-01              # 任务ID，与开发计划表一致
status: completed           # 状态：planned/in-progress/completed/blocked/cancelled
stage: P0                   # 所属阶段
title: 建立领域词汇表        # 任务标题
started_at: 2026-07-13T...  # 开始时间（ISO 8601）
completed_at: 2026-07-13T... # 完成时间
estimated_hours: 3          # 预计耗时（小时）
actual_hours: 1.5          # 实际耗时（小时）
---
```

**状态说明**：
- `planned` - 已计划，未开始
- `in-progress` - 进行中
- `completed` - 已完成并通过验收
- `blocked` - 阻塞，等待外部依赖
- `cancelled` - 已取消（需注明原因）

### 主要章节

#### 1. 目标
- 来自开发计划表的任务描述
- 明确的任务边界
- 依赖关系说明

#### 2. 验收标准
使用可检查的清单：
```markdown
- [x] 已完成项
- [ ] 未完成项
```
**原则**：标准必须可观察、可验证。

#### 3. 实现过程
记录：
- 关键步骤
- 设计决策和理由
- 遇到的问题
- 如何解决

#### 4. 修改的文件
**分类**：
- **新增文件** - 新创建的文件
- **修改文件** - 已有文件的变更
- **删除文件** - 已删除的文件

**格式**：
```markdown
### 新增文件
- `path/to/file.md` - 文件用途说明

### 修改文件
- `path/to/existing.md` - 修改内容说明

### 删除文件
- （无或列出）
```

#### 5. 测试结果
记录各类测试的通过情况：
```markdown
- ✅ 单元测试：通过（15/15）
- ✅ 集成测试：通过（8/8）
- ✅ 隐私测试：通过（9/9）
- ✅ 手动验证：通过
```

#### 6. 问题与解决
表格记录遇到的问题：

| 问题 | 解决方案 | 耗时 |
|------|---------|------|
| 描述问题 | 如何解决 | 花了多久 |

#### 7. 下一步
- **依赖任务**：哪些任务依赖本任务
- **注意事项**：后续需要注意什么

#### 8. 提交信息
- Commit hash 或 message
- PR 链接（如有）
- 评审人

## 🔄 工作流程

### 完整流程示例

```bash
# 1. 开始任务
cp development-logs/templates/task-template.md \
   development-logs/in-progress/P0-02-mvp-scope.md

# 2. 编辑日志，填写基本信息
# （task_id, title, estimated_hours 等）

# 3. 开发过程中实时更新日志
# （记录修改、问题、决策）

# 4. 完成开发，更新验收标准和测试结果

# 5. 移动到 completed
mv development-logs/in-progress/P0-02-mvp-scope.md \
   development-logs/completed/

# 6. 更新 PROGRESS.md
# （在总进度表中标记完成）

# 7. 提交代码
git add development-logs/
git commit -m "docs(dev): complete P0-02 mvp scope definition"
```

### 时间记录

**方法**：
- 在开始时记录 `started_at`
- 在完成时记录 `completed_at`
- 计算并填写 `actual_hours`

**技巧**：
- 使用时间戳追踪（`date +"%Y-%m-%dT%H:%M:%S%z"`）
- 可以分段记录，最后汇总

### 文件清单维护

**原则**：
- 每个修改的文件都要列出
- 说明修改的原因和内容
- 重要的代码变更附上关键代码片段

### 详细进度表

```markdown
| 任务ID | 任务描述 | 状态 | 开始日期 | 完成日期 | 实际人日 | 提交/PR | 备注 |
|--------|---------|------|---------|---------|---------|---------|------|
```

**状态标记**：
- ⬜ 未开始
- 🔄 进行中

- ❌ 阻塞
- 🚫 已取消

## 🎯 最佳实践

### 1. 及时更新

**推荐**：边做边记，每天至少更新一次

**避免**：一周后补记，细节丢失

### 2. 诚实记录

**推荐**：如实记录耗时和遇到的问题

**避免**：只记录成果，不记录挫折

### 3. 详细描述

**推荐**：
```markdown
## 问题与解决
| 问题 | 解决方案 | 耗时 |
|------|---------|------|
| PostgreSQL 连接池配置错误 | 修改配置使用正确的连接字符串，添加连接池健康检查 | 45分钟 |
```

**避免**：
```markdown
## 问题与解决
- 遇到了一些数据库问题，已解决
```

### 4. 链接相关文档

在日志中链接到：
- 相关 ADR
- API 文档
- 测试报告
- PR 和 Issue

### 5. 定期回顾

**每周回顾**：
- 检查进度是否符合预期
- 识别阻塞项
- 调整后续计划

**阶段结束回顾**：
- 提取经验教训
- 更新估算数据
- 优化流程

## 🔍 常见问题

### Q1：如果任务被取消怎么办？

**A**：标记为 `cancelled`，在日志中记录取消原因和替代方案。

```markdown
status: cancelled
---
## 取消原因
需求变更，功能由P10整合实现。

## 替代方案
在P10-XX中统一处理。
```

### Q2：如果任务阻塞了怎么办？

**A**：标记为 `blocked`，在 PROGRESS.md 的阻塞项表格中记录。

```markdown
status: blocked
---
## 阻塞原因
等待外部API文档

## 解除条件
收到第三方API文档

## 上报时间
2026-07-13
```

### Q3：如果实际耗时远超估计怎么办？

**A**：如实记录，分析原因，更新后续估算。

```markdown
estimated_hours: 3
actual_hours: 8
---
## 原因分析
1. 初次使用框架，学习成本高于预期
2. 遇到未文档化的bug，调试耗时较长
3. 需求理解有偏差，中途调整

## 经验总结
后续类似任务应增加 50% 缓冲时间
```

### Q4：日志需要多详细？

**A**：遵循"足够重建"原则 - 6个月后的你或同事能否根据日志理解当时发生了什么？

**足够的细节**：
- ✅ 关键决策和理由
- ✅ 遇到的问题和解决方案
- ✅ 修改的文件和原因
- ✅ 时间投入分布

**不必要的细节**：
- ❌ 每个文件的每行代码变更（用 git diff 即可）
- ❌ 语法错误和拼写错误
- ❌ 与任务无关的讨论

### Q5：如何与 Git 提交配合？

**推荐做法**：
1. 任务日志与代码提交对应
2. Commit message 中引用任务ID：`feat(p0): P0-02 freeze MVP scope`
3. PR 描述中链接任务日志

```bash
git commit -m "docs(p0): P0-02 freeze MVP scope

- Add in/out checklist
- Define acceptance criteria
- Document non-MVP features

Task: development-logs/completed/P0-02-*.md"
```
