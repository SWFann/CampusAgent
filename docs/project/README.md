# 项目状态文档

本目录只存放阶段状态、审计结论和整改计划。它回答的问题是：项目当前处在哪个阶段，哪些门禁已经通过，哪些事项仍然阻塞。

当前状态（2026-07-16）：R1-32～R1-36 已由 Codex 收口并推送至 `origin/main`；R3-25/R4-19 的远端 GitHub Actions 观察已完成且通过。后续 P2 任务交给 Claude 前，应以 `/root/CampusAgent` 为基准，先确认最新提交、工作树状态和远端 CI 状态。

## 当前权威顺序

1. [项目交接与审计工作流](PROJECT_HANDOFF_AUDIT_WORKFLOW.md)：新对话恢复上下文、任务发布、审计和提交的操作入口。
2. [P0/P1 整改计划](P0_P1_REMEDIATION_PLAN.md)：任务清单和整改顺序。
3. [P0 复审记录](P0_REVIEW_RECORD.md)：P0 契约和隐私设计的审计依据。
4. [P1 完成总结](P1_COMPLETION_SUMMARY.md)：工程底座的完成情况和剩余风险。
5. [P0 完成总结](P0_COMPLETION_SUMMARY.md)：历史总结，仅在与复审记录一致时作为参考。

## 使用规则

- Claude 或 Codex 执行任务时，以整改计划中的任务编号为最小工作单元。
- 每一次发布给 Claude、CatPaw、GLM 或其他新对话执行工具的任务指令，都必须包含足够的项目背景和当前状态，不能假设对方继承上一轮对话记忆。至少应写明：项目路径、必读文档、当前最新提交或基准状态、已完成任务、当前任务编号与目标、允许修改范围、禁止事项、自检命令、日志要求、提交/推送规则，以及后续任务不得提前执行的边界。
- 每个任务完成后，必须有验证结果和开发日志证据。
- 如果完成总结与复审记录冲突，优先相信复审记录。
- 不要在本目录放实现细节、临时代码、命令输出或长篇过程记录；这些内容放入 `development-logs/`。
