# CampusAgent 详细开发计划表

> 版本：v1.0
>
> 基线日期：2026-07-13
>
> 适用范围：比赛 MVP，从文档初始化到可离线演示的完整系统
>
> 默认执行方式：单人主导、AI 辅助、按阶段顺序交付

## 1. 如何使用这份计划

这不是按日期强行推进的甘特表，而是一套带质量门禁的执行清单。每次只把一个任务置为“进行中”，完成实现、测试、文档和验收后再勾选下一项。

状态约定：

- `[ ]` 未开始；
- `[~]` 进行中；
- `[x]` 已完成并通过验收；
- `[!]` 阻塞，需要记录原因和解除条件；
- `[-]` 经 ADR 明确取消，不允许静默跳过。

执行规则：

1. 严格按任务编号推进；标注为可并行的任务也必须在其依赖完成后开始；
2. 每个阶段先满足“进入条件”，结束前逐项检查“退出条件”；
3. 一个任务不仅包含编码，还包含测试、文档、错误处理和安全检查；
4. 任何跨模块、数据保留、外部模型或隐私边界变化都必须先写 ADR；
5. 业务代码不得直接突破完整计划书规定的模块边界；
6. 若进度不足，优先砍掉非核心展示能力，不能砍隐私控制、权限测试和清理机制；
7. 每完成一个阶段，在本文“总进度表”更新状态、完成日期和提交/PR 链接。

## 3. 全局完成定义

任意功能只有同时满足以下条件才能标为完成：

- 实现符合所属模块责任，未直接访问其他模块内部表；
- 正常、空、加载、异常、无权限和重复请求路径均有处理；
- 单元/集成测试覆盖核心规则，隐私相关功能有反向越权用例；
- 数据库变化有可回放、可回退或明确不可回退说明的迁移；
- OpenAPI、WebSocket、领域事件和 README 与实现一致；
- 日志不含密码、Token、Prompt、原始偏好、聊天明文或思维链；
- 数据的 owner、purpose、scope、consent、TTL 和删除路径清楚；
- lint、typecheck、test、build 和敏感信息检查通过；
- 关键操作有 request ID、审计结果和稳定错误码；
- 手工验收记录已填写，提交信息能对应计划任务编号。

## 5. 推荐的日常开发循环

每个计划任务按以下顺序执行：

1. 阅读任务、依赖、对应计划书章节和现有 ADR；
2. 写出 3–8 条可观察的验收条件；
3. 先补接口/Schema/迁移设计，确认数据分类和权限；
4. 写失败用例和越权用例，再实现最小功能；
5. 运行受影响测试、lint、typecheck 和契约差异；
6. 手工验证正常、空、错误、重复、撤销和超时路径；
7. 更新 API/事件/README/ADR；
8. 检查 `git diff` 中是否有密钥、真实数据或敏感日志；
9. 用一个聚焦的 Conventional Commit 提交；
10. 在本计划表勾选任务，记录提交或 PR。

推荐提交粒度：一个迁移与配套模型、一个公开契约、一个独立用例或一组紧密相关测试。避免把多个阶段压在一个巨大提交中。

1. MinIO、Prometheus/Grafana/DCGM 的真实部署，保留接口和轻量指标；
2. 外部真实模型和真实边缘 GPU 节点，保留 Mock、规则引擎和模拟指标；
3. 推荐联系人、组织复杂审批、资料导出等非演示能力；
4. 课堂、社团、学习小组、新生助手和情绪记录的展示页细节；
5. 复杂 Agent 对话和长期个性化，只保留场景所需的个人 Agent、授权和记忆；
6. Refresh Token 高级轮换可简化为安全、明确、测试过的 MVP 策略。

不可裁剪：认证与基本 RBAC、私有提交隔离、字段加密、逐场景授权、管理员拒绝、日志脱敏、不保存思维链、人工确认、TTL 清理、Mock/规则备用、核心隐私 E2E。

## 8. 立即开始的第一批任务

下一步只执行 P0，不要直接生成业务代码：

1. P0-01：建立领域词汇表；
2. P0-02：把 MVP/非 MVP 转为可验收清单；
3. P0-03：绘制宿舍聚餐用户旅程；
4. P0-04：建立角色—资源—动作权限矩阵；
5. P0-05：建立字段级数据分类与保留表；
6. P0-06/P0-07：完成数据流图和威胁模型；
7. P0-08：冻结状态机；
8. P0-09/P0-10：冻结第一版 HTTP、WebSocket 和领域事件契约；
9. P0-11：从权限矩阵生成隐私测试矩阵；
10. P0-12：用 ADR 记录并接受上述关键决策。

P0 退出评审通过后，才进入 P1 工程初始化。
