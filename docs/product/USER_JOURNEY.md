# 用户旅程：宿舍聚餐协商

> **版本**：v1.0  
> **基线日期**：2026-07-14  
> **状态**：已冻结  
> **维护者**：开发团队

## 1. 旅程概述

### 1.1 核心旅程

**宿舍聚餐去哪** - 四名学生在不暴露个人偏好的前提下完成集体决策

**旅程时长**：约 5-8 分钟（演示时长）

**涉及角色**：
- 林晓（场景发起人）
- 陈宇（宿舍成员）
- 周宁（宿舍成员）
- 许然（宿舍成员）

### 1.2 旅程阶段

```
注册登录 → 加入宿舍 → 群聊创建 → 发起场景 → 授权确认 → 私有提交
  → 系统处理 → 候选展示 → 投票确认 → 数据清理
```

#### 步骤 5.2：候选餐厅生成

**流程**：
1. 读取所有偏好胶囊
2. 从内置餐厅数据库筛选候选
3. 硬约束过滤：
   - 预算 <= min(林100, 陈80, 周120, 许150) = 80元
   - 距离 <= max(林3, 陈5, 周2, 许?) = 5公里
   - 满足饮食禁忌（不吃香菜、不吃辣）
4. 至少生成3个候选

**候选示例**：
1. **南门粤菜馆**（75元，1.2km，粤菜，支持不吃香菜）
2. **校园日料屋**（98元，0.8km，日料，支持不吃香菜）
3. **川味阁**（68元，2.5km，川菜，有不吃辣选项）
4. **湘菜馆**（72元，3.1km，湘菜）❌ 超距离

**验证点**：
- ✅ 至少3个候选
- ✅ 满足所有硬约束
- ✅ 候选排序稳定（固定种子）

---

#### 步骤 5.3：私有评分

**流程**：
1. 协调智能体向各人智能体发送候选评价请求
2. 每个智能体独立评价所有候选
3. 返回结构化评分（**不含原始偏好**）：

```json
{
  "candidate_id": "r1",
  "hard_constraint_passed": true,
  "utility_score": 0.85,
  "objection_level": "none",
  "reason_codes": ["BUDGET_OK", "CUISINE_MATCH"]
}
```

**不允许返回**：
- ❌ "用户因为经济困难只能接受80元以下"
- ❌ "用户最近焦虑所以想去安静环境"
- ❌ 用户原始自由文本
- ❌ 暴露身份的敏感理由

**验证点**：
- ✅ 每个候选都有评分
- ✅ 评分不含原始偏好
- ✅ 只返回结构化数据

---

#### 步骤 5.4：聚合评分

**聚合公式**：
```python
candidate_score = hard_constraint_gate × (
    0.60 × mean_utility          # 平均效用
  + 0.20 × fairness_score        # 公平性
  + 0.10 × distance_score        # 距离
  + 0.10 × budget_score          # 预算
)

hard_constraint_gate = 0 if any critical fails else 1
fairness_score = 1 - std(user_utility_scores)
```

**示例结果**：

| 候选餐厅 | 平均效用 | 公平性 | 距离分 | 预算分 | 总分 |
|---------|---------|--------|--------|--------|------|
| 南门粤菜馆 | 0.82 | 0.85 | 0.90 | 1.00 | **0.88** |
| 校园日料屋 | 0.75 | 0.80 | 0.95 | 0.90 | **0.82** |
| 川味阁 | 0.68 | 0.75 | 0.70 | 1.00 | **0.75** |

**验证点**：
- ✅ 使用固定公式
- ✅ 硬约束门生效
- ✅ 排序稳定可复现

- 立即清理
- 最长 24 小时兜底任务

-- 私有评价已删除
SELECT COUNT(*) FROM private_candidate_evaluations

-- 最终结果保留

**验证点**：

## 3. 异常旅程

### 3.1 授权被拒绝

**场景**：周宁拒绝授权

**流程**：
1. 周宁点击"拒绝授权"
2. 场景状态保持 WAITING_FOR_CONSENT
3. 系统发送提醒
4. 林晓可以移除周宁或取消场景

**验证点**：
- ✅ 场景不推进
- ✅ 其他成员可正常进行

---

### 3.2 模型失败降级

**场景**：模型网关不可用

**流程**：
1. 系统尝试调用模型
2. 模型节点失败
3. 降级到规则引擎
4. 使用固定规则生成结果
5. 场景正常完成

**验证点**：
- ✅ 不依赖网络
- ✅ 规则引擎产生合理结果
- ✅ 场景不崩溃

---

### 3.3 超时处理

**场景**：某成员超时未提交

**流程**：
1. 超过 TTL 时间
2. 场景推进到 CANDIDATES_READY（只包含已提交成员）
3. 在结果中标注"部分成员未提交"

**验证点**：
- ✅ 场景不无限等待
- ✅ 结果中标注未提交成员

## 4. 隐私旅程检查点

### 4.1 关键隐私控制点

| 步骤 | 隐私控制 | 验证方式 |
|------|---------|---------|
| 私有提交 | 不进入消息表 | SQL 查询 |
| 偏好胶囊 | 最小化结构化 | 字段对比 |
| 智能体评分 | 不含原始偏好 | 返回结构检查 |
| 聚合 | 不含个人身份 | 理由文本检查 |
| 候选展示 | 无个人偏好 | 人工审查 |
| 清理 | P4数据删除 | 数据库验证 |
| 审计 | 无敏感正文 | 日志检查 |

### 4.2 数据流检查

**私有偏好流**：
```
用户表单 → 加密存储 → 智能体提取胶囊 → 协调层评分 → 删除
              ↑                              ↓
         PrivateSceneSubmission      PrivateCandidateEvaluation
         (encrypted_payload)          (expires_at)
```

**不经过的路径**：
- ❌ 不经过 Message 表
- ❌ 不经过 WebSocket
- ❌ 不在日志中明文记录
