# ADR-001：采用模块化单体架构

**状态**：Accepted  
**日期**：2026-07-14  
**决策者**：开发团队

## 背景

需要选择后端架构风格。选项包括：微服务、模块化单体、单体。

## 考虑选项

### 选项A：微服务

**优点**：
- 服务独立部署
- 技术栈灵活
- 团队可并行

**缺点**：
- 运维复杂度高
- 分布式系统复杂性
- 需要服务发现、负载均衡
- 小团队难以维护

**拒绝理由**：MVP阶段团队小、运维能力有限。

---

### 选项B：模块化单体

**优点**：
- 清晰模块边界
- 低运维复杂度
- 可未来拆分为微服务
- 适合小团队并行开发

**缺点**：
- 部署耦合
- 共享数据库

**接受理由**：平衡开发效率和架构合理性。

---

### 选项C：单体

**优点**：
- 最简单
- 无跨模块依赖

**缺点**：
- 代码混乱风险高
- 难以维护
- 未来拆分困难

**拒绝理由**：缺乏模块边界约束，长期难以维护。

---

## 决策

**采用模块化单体**。

## 后果

- 所有业务模块放在 `backend/app/modules/` 下
- 每个模块有独立目录：`api.py`, `schemas.py`, `models.py`, `repository.py`, `service.py`
- 模块间只能通过公开 Service Interface 和 Schema 调用
- 禁止直接导入其他模块的 ORM Model
- 通过领域事件解耦非同步行为
- 未来可通过提取模块拆分为微服务
