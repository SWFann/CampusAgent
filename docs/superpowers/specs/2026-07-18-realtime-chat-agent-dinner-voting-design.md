# CampusAgent 实时群聊与智能体聚餐投票设计

## 1. 目标

本设计升级 CampusAgent 的群聊与宿舍聚餐场景，交付以下能力：

- 群消息和场景状态对在线接收方实时可见，断线后可自动补齐；
- 宿舍聚餐以群聊中的可展开投票消息存在，不固定占据聊天页面；
- 每位参与者的智能体根据其私密偏好独立联网检索真实餐厅；
- 聚餐协调智能体基于有来源的搜索结果完成多轮汇总和候选排序；
- 群成员可查看脱敏后的详细智能体辩论过程，并对真实候选投票；
- 出现反对意见后，可继承场景记忆发起下一次协商；
- 保持现有隐私边界，禁止公开或记录成员原始私密偏好。

## 2. 非目标

- 不接入第三方地图、点评或支付平台；
- 不实现订座、导航、支付或到店核销；
- 不让模型在没有搜索证据时生成餐厅；
- 不把场景临时记忆自动写入长期个人记忆；
- 不重新设计通用聊天产品的全部界面。

## 3. 核心原则

### 3.1 服务端状态权威

聊天消息、场景状态、投票、辩论进度和最终结果均以服务端持久化状态为准。WebSocket 用于低延迟通知，HTTP 快照与增量查询用于恢复和校验。

### 3.2 真实证据先于模型推荐

真实餐厅候选先由 StepFun Search API 获取网页证据，再由 `step-3.7-flash` 基于证据筛选和解释。模型不得凭参数知识或已有记忆补造餐厅、地址、价格或营业信息。

### 3.3 私密输入与公开辩论隔离

成员的预算、饮食禁忌和补充描述只在本人私有域中处理。智能体对外只发布经过校验的脱敏立场。即使演示模式展示“张三Agent”，也不得展示张三的原始输入。

### 3.4 失败时宁缺毋滥

搜索失败、来源不足或模型输出无法验证时，系统显示明确错误并允许重试。系统不得回退到写死或虚构餐厅。

## 4. 群聊实时通信

### 4.1 写入与推送链路

所有普通消息和场景公共消息统一通过 ConversationService 写入：

```text
Command
→ validate permission and idempotency key
→ persist message/state in one transaction
→ commit
→ publish domain event
→ Redis Pub/Sub
→ WebSocket subscribers
```

场景服务不得直接创建 `Message` ORM 记录。它通过 ConversationService 的公共接口发布 `SCENE_CARD`、`AGENT_PUBLIC`、`VOTE` 和 `RESULT` 消息，从而保证事务提交后一定进入实时事件链路。

### 4.2 事件契约

每个实时事件至少包含：

- `event_id`：全局唯一，用于去重；
- `event`：事件类型；
- `conversation_id`：群聊标识；
- `sequence`：群聊内单调递增序号；
- `resource_id`：消息或场景标识；
- `resource_version`：场景快照版本；
- `timestamp`：服务端产生时间。

事件正文不携带私密偏好。客户端收到 `message.created` 后按消息 ID 合并；收到 `scene.updated` 后获取或合并最新公共场景快照。

### 4.3 恢复与降级

- 客户端按 `event_id` 去重并按 `sequence` 排序；
- 发现序号缺口时立即执行 HTTP 增量回补；
- WebSocket 重连、浏览器页面从后台恢复时重新校验最新 sequence 和场景版本；
- WebSocket 长时间不可用时启用有限频率 HTTP 轮询，恢复后切回实时推送；
- Redis Pub/Sub 失败不回滚已经成功提交的业务事务，客户端通过 HTTP 恢复；
- 发送消息、投票、启动协商、关闭投票和结束场景都携带幂等键；
- 场景写入使用 `resource_version` 做乐观并发控制，旧响应不得覆盖新状态。

### 4.4 耗时任务

联网搜索和多智能体辩论在后台任务中执行。API 快速返回任务标识，后端持续发布以下公共进度：

- 正在构建匿名搜索条件；
- 成员 Agent 正在检索；
- Agent 已完成第 N 轮；
- 聚餐协调 Agent 正在汇总；
- 候选已生成或任务失败。

同一场景同一时刻只允许一个活动协商任务。关闭投票或结束聚餐后，未完成任务被取消，迟到结果不得写回。

## 5. 聊天原生聚餐投票消息

### 5.1 发起入口

群聊输入区提供“宿舍聚餐”工具按钮。点击后弹出创建窗口，要求填写：

- 城市；
- 校区或出发地点；
- 聚餐主题；
- 投票截止时间；
- 首次协商的辩论轮数，范围 1–10。

确认后，系统在群聊中发送一条 `SCENE_CARD` 消息。聚餐场景不固定占据聊天页面。

### 5.2 折叠卡片

群聊消息中的折叠卡片显示：

- 发起人和聚餐主题；
- 当前阶段；
- 参与、跳过和已提交人数；
- 当前协商次数和辩论轮次；
- 投票截止时间；
- “参与协商”和“打开投票”操作。

卡片打开时读取最新场景快照，不能依赖历史消息中的旧正文。

### 5.3 详情弹窗

详情弹窗包含四个页签：

1. **我的需求**：预算、时间、菜系、饮食限制、距离、环境和私密补充描述；
2. **智能体协商**：逐轮展示各成员 Agent 的检索摘要、候选主张、支持或反对理由，以及协调 Agent 的合并、淘汰和排序过程；
3. **候选投票**：展示真实候选、匹配信息、来源证据和投票操作；
4. **最终结果**：展示票数、智能体综合评分、共识、未解决分歧和最终建议。

智能体名称默认匿名为“成员A Agent”“成员B Agent”。演示模式可显示“张三Agent”等姓名，但该切换只影响显示名称，不放宽私密数据规则。总协调者命名为“聚餐协调Agent”。

## 6. 私密需求与搜索条件

### 6.1 成员输入

每位参与者单独提交：

- 预算范围；
- 可用时间；
- 菜系偏好；
- 饮食限制；
- 可接受距离；
- 环境偏好；
- 私密补充需求。

补充需求只对本人及其 Agent 可见。群内只展示匿名、脱敏后的公共观点。

### 6.2 最小化外部输入

调用外部搜索和模型前，系统将私密需求转换为匿名、完成任务所需的最小条件。外部请求不得包含用户 ID、姓名、联系方式、精确经济标签或可指认的原始自由文本。

## 7. 真实搜索与模型处理

### 7.1 两阶段提供者

使用同一个 StepFun API Key，但保持两个独立职责：

1. `POST https://api.stepfun.com/v1/search` 获取互联网搜索结果；
2. `step-3.7-flash` 根据搜索证据生成结构化候选与公开说明。

API Key 仅从 `MODEL_GATEWAY_API_KEY` 或等价的服务端 Secret 配置读取，不进入浏览器、数据库、日志、测试固件或 Git。

### 7.2 成员 Agent 流程

每位参与成员拥有一个本场景 Agent。每轮中，该 Agent：

1. 根据自己的私密偏好和场景内记忆生成搜索策略；
2. 使用城市、校区/出发地点和匿名条件搜索真实地点；
3. 从搜索结果中选择有证据支持的候选；
4. 生成脱敏的公开主张、支持理由和反对理由；
5. 引用支撑其主张的来源结果。

### 7.3 聚餐协调 Agent 流程

协调 Agent 只能读取成员 Agent 的脱敏输出和已验证来源。它负责：

- 合并同名或明显重复的地点；
- 排除地点不符、证据不足或冲突无法解释的候选；
- 比较群体约束、公平性、距离和预算适配；
- 记录候选排序变化和淘汰原因；
- 产出本轮公共总结和下一轮关注点。

### 7.4 候选证据

每个候选至少保存并展示：

- 餐厅名称；
- 地址或区域线索；
- 价格线索；
- 营业信息线索；
- 匹配标签；
- 推荐理由与风险提示；
- 来源网页标题；
- 可点击的 HTTPS 来源 URL；
- 检索时间。

地址、价格或营业时间缺少充分来源时显示“未核实”。卡片固定提示“信息可能变化，请到店前确认”。没有有效来源的候选不得进入投票。

## 8. 多次协商、投票与记忆

### 8.1 轮次设置

每次启动协商时，发起人设置本次辩论轮数，取值范围 1–10。下一次协商可以选择不同轮数，并继承此前的场景内记忆。

### 8.2 投票选项

参与者可以：

- 同意一个候选；
- 选择“都不同意”并填写私密原因；
- 请求下一次协商。

出现反对或下一次协商请求时，不自动伪造共识。发起人决定是否设置新的轮数并启动下一次协商。

### 8.3 场景内记忆

场景内记忆包括：

- 历次搜索词、来源和检索时间；
- 各 Agent 的脱敏立场及变化；
- 历次候选、排序和淘汰理由；
- 协调 Agent 的轮次总结；
- 投票结果和反对原因的匿名摘要；
- 下一次协商需要解决的分歧。

记忆用于避免重复搜索、跟踪分歧和指导下一次协商。场景结束后默认清理。只有用户明确二次确认后，才可将其本人的长期偏好写入个人记忆。

## 9. 权限与生命周期

### 9.1 权限

- 群成员可以参与或退出协商、管理自己的私密需求、查看公共辩论、投票、反对并请求下一次协商；
- 只有发起人可以启动协商、设置本次轮数、关闭投票或结束聚餐；
- 非群成员不能读取或操作场景；
- 发起人、群管理员和系统管理员均不能读取其他成员的原始私密需求；
- 身份显示模式不得改变数据访问权限。

### 9.2 状态流

```text
DRAFT
→ COLLECTING_PARTICIPATION
→ COLLECTING_PRIVATE_INPUT
→ RESEARCHING
→ DEBATING
→ VOTING
→ DISAGREEMENT
   → RESEARCHING (发起人启动下一次协商)
   → VOTING
→ VOTING_CLOSED
→ COMPLETED
```

`FAILED`、`CANCELLED` 和 `EXPIRED` 为异常或提前终态。关闭投票停止接收新票并保留公共结果；结束聚餐停止所有活动任务并执行临时私密数据清理。

## 10. 错误处理

- 搜索调用失败：记录不含正文和密钥的元数据，展示可重试错误，不生成候选；
- 搜索结果不足：允许候选少于预期，不补造候选；
- 模型调用失败：保留已验证搜索证据，允许重新汇总；
- 模型结构错误：严格 Schema 校验并做有限重试，仍失败则标记任务失败；
- 来源 URL 无效：剔除对应证据；候选无剩余证据时一并剔除；
- 并发版本冲突：客户端刷新最新快照后重新操作；
- 任务被关闭或场景结束：停止后续步骤并拒绝迟到写入。

## 11. 测试与验收

### 11.1 后端单元测试

- 权限矩阵与仅发起人操作；
- 状态转换与每次 1–10 轮校验；
- 私密输入最小化和公开内容泄露检测；
- 搜索结果与候选来源关联校验；
- 模型结构化输出、重试和失败路径；
- 场景内记忆继承、结束清理和长期记忆二次确认；
- 幂等消息、幂等投票和乐观并发。

### 11.2 集成测试

- 消息事务成功后发布实时事件；
- 两个浏览器会话实时接收普通消息和场景更新；
- 断线重连、sequence 缺口、重复与乱序事件恢复；
- WebSocket 降级轮询和恢复；
- 多用户并发投票；
- 关闭投票或结束场景时取消运行中的协商；
- Redis 短暂不可用后的 HTTP 状态恢复。

### 11.3 模型与搜索测试

自动化测试使用 `httpx.MockTransport` 或项目既有等价测试替身，不调用真实服务。手工验收使用已经轮换的 StepFun Key，验证 `/v1/search` 和 `step-3.7-flash`，并逐一打开候选来源 URL。

### 11.4 前端测试

- 群聊工具按钮和创建弹窗；
- 折叠场景消息与四页签详情弹窗；
- 匿名/演示名称切换但数据内容不变；
- 辩论实时进度和多次协商历史；
- 投票、反对、关闭和结束权限；
- 加载、失败、重试、离线和恢复状态。

### 11.5 质量门禁

实现完成后执行仓库既有 Python 与 Web 的 lint、typecheck、test 和 build 检查。任何无法运行或失败的检查必须如实报告，不能用真实外部调用替代自动化测试。

## 12. 安全说明

用户曾在需求对话中提供一个真实 API Key。该 Key 应在 StepFun 控制台撤销并轮换。设计和实现不得复制、存储或使用对话中暴露的 Key，只接受由部署环境注入的新 Key。
