# 比赛 Demo 规范

## 演示目标

在约 5 分钟内证明：CampusAgent 能让四名学生在不互相公开预算、禁忌和私人理由的前提下完成一次宿舍聚餐决策，并让用户看见授权、审计和数据清理能力。

## 固定演示角色

所有角色均为虚构数据：

| 角色 | 用途 | 私有偏好示例（不得在公共界面出现） |
|---|---|---|
| 林晓 | 场景发起人 | 偏好安静、距离近 |
| 陈宇 | 宿舍成员 | 预算上限较低 |
| 周宁 | 宿舍成员 | 有饮食禁忌 |
| 许然 | 宿舍成员 | 偏好日料、时间受限 |
| 校方管理员 | 展示模型/节点 | 只能看运行指标 |

演示数据不得映射真实同学或真实学号。

## 必备页面/状态

1. 登录与四名学生账号切换；
2. 宿舍组织详情与四人群聊；
3. 聚餐场景发起卡和参与确认；
4. 每名用户独立的私有偏好表单；
5. 群聊中的提交进度与隐私说明；
6. 三项候选结果卡、聚合理由和投票；
7. 最终确认与“是否写入长期记忆”询问；
8. 记忆访问记录和临时数据清理结果；
9. 不包含个人偏好的模型/边缘节点管理视图。

## 标准演示流程

1. 用林晓发起“周五宿舍聚餐”；
2. 展示参与者必须逐人确认 L2 场景授权；
3. 切换四个学生账号分别提交私有表单；
4. 回到群聊，确认界面只显示 `3/4`、`4/4` 进度；
5. 启动处理，展示模型网关选择本地或 Mock 路径；
6. 展示三项候选与“满足全部硬性限制”等聚合理由；
7. 投票并由发起人确认结果；
8. 拒绝写入长期记忆，展示临时数据已清理；
9. 打开审计页，只显示某场景在某目的下读取了某类偏好；
10. 打开管理员视图，证明没有查看个人偏好的入口。

## 候选数据要求

- 至少 8 家虚构或明确授权使用的餐厅；
- 字段包括菜系、均价、距离、环境标签和支持的饮食限制；
- 演示全流程不依赖公网地图或外部餐厅 API；
- 固定种子保证至少 3 项有效候选和可复现排名。

## 验收清单

- [ ] A 用户无法查看 B 的原始提交；
- [ ] 群主无法查看任何成员自由文本；
- [ ] 普通消息历史没有私有偏好；
- [ ] 群聊没有智能体辩论或思维链；
- [ ] 公共理由不指认具体成员；
- [ ] 未授权成员不能提交或触发处理；
- [ ] 候选、投票和最终确认可完成；
- [ ] 场景结束后 P4 数据被清理；
- [ ] 用户可查看结构化访问记录；
- [ ] 管理员页面不提供私有内容入口；
- [ ] 断网或模型失败时可走规则/Mock 备用路径；
- [ ] 授权或加密服务失败时拒绝执行。

## 演示范围控制

课堂、社团、学习小组、新生助手和情绪记录只展示场景卡与设计说明，不在比赛 Demo 中伪装为已完成能力。
